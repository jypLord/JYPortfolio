# AutoInvest 차트 수신 방식 정리

## 결론

백엔드가 `15:30(KST) 이후 차트를 단 한 번만 반환`하도록 바뀌었다면, 프런트는 `SSE`보다 `fetch`가 더 적합하다.

이번 프로젝트에서는 다음 방식이 가장 효율적이다.

1. 프런트가 `15:30(KST)` 이전에는 요청을 보내지 않는다.
2. `15:30(KST)`가 되면 종목별로 `fetch`를 한 번 실행한다.
3. 응답 데이터를 차트에 렌더링하고 연결은 즉시 종료한다.

## CS 관점 이유

### 1. 통신 모델 일치

SSE는 `서버가 여러 번 push`하는 상황에 최적화되어 있다.
하지만 지금 요구사항은 `서버가 한 번만 응답`하는 모델이다.

즉, 현재 문제는 본질적으로 `streaming`이 아니라 `single response request`다.
통신 모델이 단건 응답이면 단건 요청인 `fetch`가 더 자연스럽다.

### 2. 서버 리소스 효율

SSE는 연결을 열린 상태로 유지한다.
사용자가 많아질수록 서버는 다음 리소스를 더 오래 붙잡게 된다.

- TCP 연결
- 소켓/파일 디스크립터
- 연결 상태 메모리
- 이벤트 루프 또는 스레드 관리 비용

반면 `fetch`는 요청-응답이 끝나면 바로 종료된다.
단건 응답에서는 연결 유지 비용을 낼 이유가 거의 없다.

### 3. 장애 처리 단순화

SSE는 다음 처리가 추가로 필요해진다.

- 재연결 정책
- 중복 이벤트 방지
- 연결 종료 시점 정의
- 네트워크 단절 시 상태 복구

한 번만 받는 데이터에 이런 복잡도를 넣으면 구현 대비 이득이 작다.
`fetch`는 성공/실패/취소만 다루면 되므로 프런트 상태 관리가 훨씬 단순하다.

### 4. 브라우저/프록시 친화성

SSE는 중간 프록시, 로드밸런서, 타임아웃 정책의 영향을 더 받는다.
특히 장시간 유휴 연결은 인프라 설정에 따라 예기치 않게 끊길 수 있다.

`fetch`는 일반적인 HTTP 요청이므로 CDN, 프록시, 로드밸런서 환경에서 다루기 쉽다.
배포 후 운영 안정성도 보통 더 높다.

### 5. 확장성 있는 비용 구조

현재 요구사항은 하루 1회 수신이다.
이 패턴은 대부분의 시간을 `대기`가 차지한다.

이 대기를 서버 연결로 처리하면 서버 비용이 늘어난다.
이 대기를 클라이언트 타이머로 처리하면 서버는 실제 필요한 순간에만 일한다.

즉, "기다림"을 서버가 아니라 클라이언트가 맡는 쪽이 더 싸다.

## 언제 SSE가 다시 맞는가

다음과 같이 바뀌면 SSE를 다시 고려할 수 있다.

- 장중에 차트가 지속적으로 갱신됨
- 가격 변동이 수초 또는 수분 단위로 push됨
- 사용자가 새로고침 없이 실시간 변화를 봐야 함
- 서버가 이벤트 스트림 자체를 핵심 계약으로 제공함

즉, `여러 번 밀어줘야 할 때` SSE가 맞다.

## 이번 프런트 수정 내용

기존에는 `EventSource`로 차트 스트림을 열었다.
현재는 다음으로 변경했다.

1. `15:30(KST)` 이전에는 요청을 보내지 않고 예약 상태 메시지만 노출
2. `15:30(KST)` 이후에는 `fetch`로 차트 데이터를 한 번 조회
3. 조회 성공 시 차트 렌더링
4. 컴포넌트가 사라지면 `AbortController`로 요청 취소

## 구현상 장점

- 서버 연결 유지 비용 제거
- 프런트 코드 단순화
- 에러 처리 단순화
- 배포 환경에서 타임아웃 이슈 감소
- 현재 백엔드 계약과 정확히 일치

## 주의할 점

프런트가 `Asia/Seoul` 기준 `15:30`을 계산하므로, 사용자의 로컬 타임존과 무관하게 동일한 기준으로 동작해야 한다.

백엔드가 앞으로 다시 실시간 전송으로 바뀌면, 그때는 `fetch polling`과 `SSE` 중 어느 방식이 더 적합한지 다시 판단하면 된다.
